# 02 测试计划

## 测试目的

通过对 OpenCart 电商系统进行测试，验证其核心业务流程是否能够正常完成，发现功能、交互、边界值和异常流程中的问题，并形成完整测试文档。在手工测试基础上，补充 Web 自动化测试和 Postman 接口测试设计，用于覆盖高频回归场景和前后端接口交互。

## 测试策略

- 优先覆盖主业务流程：注册登录 -> 商品浏览 -> 加入购物车 -> 下单结算 -> 后台查看订单
- 其次覆盖后台维护流程：商品新增/编辑、订单状态处理、客户管理
- 每个核心模块至少覆盖：
  - 正向流程
  - 必填项校验
  - 边界值
  - 异常输入
  - 权限或状态限制
- Web 自动化测试优先覆盖稳定、高频、重复执行价值高的流程
- Postman 接口测试优先覆盖登录、商品查询、购物车、订单相关接口

## 测试方法

| 方法 | 应用场景 |
| --- | --- |
| 等价类划分 | 用户名、邮箱、密码、商品数量等输入项 |
| 边界值分析 | 购物车数量、价格、库存、文本长度 |
| 场景法 | 注册、登录、下单、后台处理订单等业务链路 |
| 错误推测法 | 重复提交、空输入、非法字符、未登录访问 |
| 流程分析法 | 订单状态流转、商品上下架、客户审批 |
| Web 自动化回归 | 登录、商品搜索、购物车、下单主流程 |
| 接口测试 | 登录、商品查询、购物车、订单等接口的状态码、响应字段和业务规则 |

## 入口准则

- 测试环境可访问
- 前台和后台基础功能可进入
- 至少准备一个普通用户账号和一个管理员账号
- 测试文档模板已建立

## 出口准则

- 核心测试用例设计完成
- 主流程用例执行完成
- 缺陷或风险点已记录
- Web 自动化测试补充范围已整理
- Postman 接口测试补充范围已整理
- 测试总结已完成
- GitHub/Gitee 链接可访问

## 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 本地环境搭建耗时 | 影响测试执行 | 可先使用在线演示环境或截图补充说明 |
| 实际缺陷较少 | 缺陷报告不够充实 | 记录体验问题、边界问题、疑似问题和风险点 |
| 时间紧张 | 文档不完整 | 优先保证 README、测试计划、用例、缺陷、总结 |
| GitHub/Gitee 推送失败 | 无法提交链接 | 至少保留本地仓库与压缩包，优先排查远程配置 |
| OpenCart 接口不清晰 | Postman 用例难以直接执行 | 先通过浏览器开发者工具抓取请求，再整理接口测试集合 |
| 自动化脚本不稳定 | 回归结果不可信 | 优先选择页面结构稳定的主流程，减少对易变元素的依赖 |

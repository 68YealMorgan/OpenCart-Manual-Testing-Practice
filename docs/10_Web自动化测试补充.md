# 10 Web 自动化测试补充

## 补充目的

在手工测试完成业务理解和用例设计后，选择稳定、高频、重复执行价值高的流程进行 Web 自动化回归测试设计。自动化测试的目标不是替代手工测试，而是提高核心流程回归效率。

## 适合自动化的流程

| 流程 | 是否建议自动化 | 原因 |
| --- | --- | --- |
| 用户登录 | 是 | 登录是多数业务流程的前置条件 |
| 商品搜索 | 是 | 搜索入口稳定，断言明确 |
| 商品详情页加购 | 是 | 连接商品浏览和购物车 |
| 修改购物车数量 | 是 | 适合校验数量和金额变化 |
| 下单主流程 | 是 | 覆盖电商核心业务闭环 |
| 后台商品新增 | 谨慎 | 表单字段较多，维护成本较高 |
| 页面样式检查 | 不优先 | 视觉变化频繁，不适合作为初期重点 |

## 自动化用例清单

| 编号 | 用例名称 | 关键断言 |
| --- | --- | --- |
| AUTO_LOGIN_001 | 普通用户登录成功 | 页面进入用户中心或出现用户菜单 |
| AUTO_LOGIN_002 | 错误密码登录失败 | 页面出现错误提示，不进入用户中心 |
| AUTO_SEARCH_001 | 搜索存在商品 | 搜索结果包含目标商品 |
| AUTO_SEARCH_002 | 搜索不存在商品 | 页面提示无匹配商品 |
| AUTO_CART_001 | 商品加入购物车 | 购物车数量增加，购物车中出现商品 |
| AUTO_CART_002 | 修改购物车数量 | 数量更新成功，总价变化正确 |
| AUTO_CART_003 | 删除购物车商品 | 商品行消失或购物车为空 |
| AUTO_CHECKOUT_001 | 登录用户完成下单主流程 | 出现订单成功提示或订单号 |

## 脚本设计建议

- 使用独立测试数据，避免污染真实账号。
- 把登录、打开商品、清空购物车等重复动作封装成公共方法。
- 页面定位优先使用稳定的 ID、name、data 属性，其次再考虑 CSS 选择器。
- 每条用例执行失败时保存截图，便于复盘。
- 本地账号、密码、地址等敏感数据不要提交到仓库。

## 可以放进简历的表达

在 OpenCart 电商测试项目中，基于手工测试用例筛选登录、商品搜索、购物车和下单等核心流程，设计 Web 自动化回归测试场景，关注脚本可维护性、断言有效性和失败截图留存。
